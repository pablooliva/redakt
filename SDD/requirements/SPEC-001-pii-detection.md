This specification defines the PII Detection endpoint (`POST /api/detect`) and the foundational project scaffolding for Redakt. This is the first feature implemented and establishes the tech stack, project structure, Docker Compose configuration, and Presidio integration patterns used by all subsequent features.

## Research Foundation

### Critical Review Findings Incorporated

- Switched from transformers to spaCy multilingual (EN + DE support)
- Lowered score threshold from 0.7 to 0.35 (avoids silent false negatives)
- Switched from langdetect to lingua-py (reliable on short text)
- Added language validation before forwarding to Presidio
- Added startup readiness handling
- Defined request size limits (500KB text, 20MB documents)

- Presidio Analyzer `POST /analyze`: `presidio/presidio-analyzer/app.py:65-116`

## Intent

### Problem Statement

Users in a German enterprise need a simple way to check whether text contains PII before pasting it into AI tools. Presidio's `/analyze` endpoint returns detailed entity arrays — users and AI agents need a boolean answer with optional detail.

### Solution Approach

Build a FastAPI wrapper that:
1. Auto-detects language (or accepts explicit override)
2. Validates and forwards to Presidio's Analyzer
3. Reduces the response to a boolean + summary
4. Logs metadata for compliance auditing

This is also the scaffolding milestone — establishing the project structure, Docker Compose, and Presidio client that all other features build on.

### Expected Outcomes

- `POST /api/detect` returns `has_pii: true/false` with entity summary
- `GET /api/health` reports readiness of Redakt and Presidio services
- `docker compose up --build` brings up the entire stack (Redakt + Presidio)
- Web UI page at `/detect` for manual PII checking
- OpenAPI docs auto-generated at `/docs`

## Success Criteria

### Functional Requirements

- REQ-001: `POST /api/detect` accepts JSON with `text` (required), `language` (default `"auto"`), `score_threshold` (default `0.35`), `entities` (optional), `allow_list` (optional)
- REQ-002: When `language` is `"auto"`, Redakt detects language using lingua-py and passes the resolved code to Presidio. Falls back to `"en"` if detection fails.
- REQ-003: Response includes `has_pii` (bool), `entity_count` (int), `entities_found` (list of unique entity type strings), `language_detected` (string)
- REQ-004: When `?verbose=true` query param is set, response additionally includes `details` — an array of `{entity_type, start, end, score}` objects
- REQ-005: Instance-wide allow list (from config) is merged with per-request `allow_list` before forwarding to Presidio. Matching is case-sensitive.
- REQ-006: Every request is audit-logged with metadata only: timestamp, action, entity_count, entity_types, language, source (web_ui or api). Never log input text or PII values. Source is determined by the `HX-Request` header — if present, source is `web_ui`, otherwise `api`.
- REQ-007: `GET /api/health` returns `{"status": "healthy", "presidio_analyzer": "up", "presidio_anonymizer": "up"}` or appropriate degraded status
- REQ-008: Web UI page at `/detect` provides a text area, a language toggle (Auto / English / German), and displays detection results via HTMX. The HTMX form submits to `/detect/submit` (returns HTML fragment), separate from `/api/detect` (returns JSON for API clients/agents).
- REQ-009: `docker compose up --build` starts Redakt, Presidio Analyzer (spaCy multilingual), and Presidio Anonymizer as three services

### Non-Functional Requirements

- PERF-001: Detect endpoint responds within 3 seconds for text under 10KB (excluding Presidio cold start)
- SEC-001: Text input capped at 500KB via Pydantic validation. Requests exceeding this return 422.
- SEC-002: Presidio services are on an internal Docker network, not exposed to the host
- SEC-003: No PII is persisted or logged at any point in the detect flow
- UX-001: Web UI shows a three-option language toggle: Auto (default), English, German. When Auto is selected, the detected language is shown in the results. Users can switch to English or German and resubmit to override detection. Only English and German are supported.
- UX-002: OpenAPI interactive docs available at `/docs` (auto-generated by FastAPI)

## Edge Cases

### EDGE-001: Empty text

- Desired behavior: Return `{"has_pii": false, "entity_count": 0, "entities_found": [], "language_detected": "unknown"}`
- Test approach: Submit `{"text": "", "language": "auto"}`

### EDGE-002: Text with no PII

- Desired behavior: Return `has_pii: false` with empty entities
- Test approach: Submit `{"text": "The weather is nice today"}`

### EDGE-003: Unsupported language detected or provided

- Desired behavior: Return HTTP 400 with `{"detail": "Language 'ja' is not supported. Supported languages: en, de"}`
- Test approach: Submit `{"text": "...", "language": "ja"}`

### EDGE-004: Language auto-detection failure

- Desired behavior: Fall back to `"en"`, include `language_detected: "en"` in response
- Test approach: Submit very short or ambiguous text (e.g., `{"text": "123"}`)

### EDGE-005: Allow list suppresses all detections

- Desired behavior: Return `has_pii: false` — allow list filtering happens in Presidio, Redakt sees an empty result
- Test approach: Submit text with PII that exactly matches the allow list entries

### EDGE-006: Very long text (near 500KB limit)

- Desired behavior: Processes successfully, may take longer. Response time not guaranteed under 3 seconds for large text.
- Test approach: Submit 400KB of text with scattered PII

### EDGE-007: Text exceeding 500KB

- Desired behavior: Return HTTP 422 with Pydantic validation error
- Test approach: Submit >512,000 character text

### EDGE-008: Presidio becomes unhealthy after initial startup

- Desired behavior: Same as FAIL-001 — return HTTP 503. Health endpoint reflects degraded state.
- Test approach: Stop Presidio container after Redakt is running, submit request

### EDGE-009: Score threshold set to 0.0

- Desired behavior: Allowed. Returns every possible detection including low-confidence noise. May produce false positives.
- Test approach: Submit `{"text": "...", "score_threshold": 0.0}` — verify more results than default threshold

### EDGE-010: Mixed-language text (German + English)

- Known v1 limitation: lingua-py returns one language. Presidio runs recognizers for that language only. Language-specific formats from the other language may be missed.
- Mitigating factor: Pattern-based recognizers (email, credit card, IBAN, IP address, URL) are language-agnostic and fire regardless of the language parameter.
- Test approach: Submit mixed text, verify pattern-based entities are still detected

## Failure Scenarios

### FAIL-001: Presidio Analyzer unavailable

- Trigger: Presidio container not running, crashed, or still starting up
- Expected behavior: Return HTTP 503 with `{"detail": "Presidio Analyzer service is unavailable"}`
- User communication: Web UI shows "Service is starting up, please wait..." if health check fails
- Recovery: Redakt retries Presidio health check on next request. No crash.

### FAIL-002: Presidio returns unexpected error

- Trigger: Presidio returns HTTP 500 for an internal error
- Expected behavior: Return HTTP 502 with `{"detail": "PII detection service returned an error"}`
- User communication: Web UI shows generic error message, audit log captures the failure
- Recovery: No retry. User can resubmit.

### FAIL-003: Language detection timeout

- Trigger: lingua-py takes too long on very large text
- Expected behavior: Fall back to `"en"` after 2-second timeout
- Recovery: Automatic fallback, no user action needed

### FAIL-004: httpx connection timeout to Presidio

- Trigger: Network issue or Presidio overloaded
- Expected behavior: httpx timeout set to 30 seconds. On timeout, return HTTP 504 with `{"detail": "PII detection service timed out"}`
- Recovery: User can retry

## Implementation Constraints

### Technical Constraints

- Presidio Analyzer requires an explicit `language` parameter — Redakt must always resolve before forwarding
- Presidio returns HTTP 500 for unsupported languages (not 400) — Redakt must validate first
- Allow list matching is case-sensitive in exact mode — documented, not changed
- spaCy multilingual config supports `en` and `de` only — other languages will fail validation
- German model upgraded to `de_core_news_lg` (from default `de_core_news_md`) for better NER accuracy. Requires modifying `spacy_multilingual.yaml` or providing a custom config.

### Docker Compose Configuration

```yaml
services:
  redakt:
    build:
      context: .
      dockerfile: Dockerfile
    ports:
      - "8000:8000"
    environment:
      - PRESIDIO_ANALYZER_URL=http://presidio-analyzer:5001
      - PRESIDIO_ANONYMIZER_URL=http://presidio-anonymizer:5001
    depends_on:
      presidio-analyzer:
        condition: service_healthy
      presidio-anonymizer:
        condition: service_healthy

  presidio-analyzer:
    build:
      context: ./presidio/presidio-analyzer
      args:
        - NLP_CONF_FILE=presidio_analyzer/conf/spacy_multilingual.yaml
    environment:
      - PORT=5001
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:5001/health"]
      interval: 10s
      timeout: 5s
      retries: 10
      start_period: 30s

  presidio-anonymizer:
    build:
      context: ./presidio/presidio-anonymizer
    environment:
      - PORT=5001
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:5001/health"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 10s
```

Notes:
- Both Presidio services listen on port 5001 internally (overriding the Dockerfile default of 3000). They are accessed by service name on the Docker network — no port mapping to host needed. Only Redakt exposes port 8000.
- The NLP config uses a custom `spacy_multilingual.yaml` with `de_core_news_lg` (upgraded from the default `de_core_news_md`). This file should be created at the project root (e.g., `config/spacy_multilingual.yaml`) and mounted or copied into the build context.

### Identified Risks

- RISK-001: **spaCy NER accuracy for German names may be lower than transformers.** Mitigation: Monitor detection quality in production. Upgrade path to multilingual transformer is documented. Regex-based German recognizers (tax ID, passport, etc.) are unaffected.
- RISK-002: **lingua-py adds ~30MB to Docker image.** Mitigation: Acceptable tradeoff for accuracy. Can switch to lighter library later if size becomes an issue.
- RISK-003: **Presidio startup time may exceed health check retries.** Mitigation: `start_period: 30s` with 10 retries at 10s intervals gives Presidio up to 130 seconds to start. Transformer models would need more — but we're using spaCy which is faster.

### Key Implementation Details

**FastAPI lifespan**: Use lifespan context manager to create the httpx client on startup and probe Presidio's health + cache supported languages.

```python
@asynccontextmanager
async def lifespan(app: FastAPI):
    app.state.http_client = httpx.AsyncClient(timeout=30.0)
    # Probe Presidio and cache supported languages
    app.state.supported_languages = await get_supported_languages(app.state.http_client)
    yield
    await app.state.http_client.aclose()
```

**Audit logging**: Use Python's `logging` module with a JSON formatter writing to stdout. Docker captures stdout logs automatically.

**HTMX pattern for detect page**: Two separate routes:
- `/api/detect` — returns JSON. Used by AI agents and API clients.
- `/detect/submit` — returns an HTML fragment. Used by the HTMX form (`hx-post="/detect/submit"` with `hx-target="#results"`). Internally calls the same detection logic, then renders a Jinja2 partial with the results.

This cleanly separates API and UI concerns. Audit logging uses the `HX-Request` header to tag source: requests from `/detect/submit` (which always have `HX-Request: true`) are logged as `web_ui`, all others as `api`.

**Audit logging**: Use Python's `logging` module with two loggers:
- `redakt.audit` — INFO level, JSON formatter, writes to stdout. Captures compliance metadata only.
- `redakt` — WARNING level by default (configurable via `LOG_LEVEL` env var). Application debug/error logs.

### Implementation Insights
1. Presidio's `allow_list` is a simple top-level request parameter — no ad_hoc_recognizer wrappers needed
2. FastAPI `response_model` with union types can strip fields from subclass responses — better to omit `response_model` when returning polymorphic types

4. Volume-mounted source in Docker requires configurable base_dir for template/static resolution
