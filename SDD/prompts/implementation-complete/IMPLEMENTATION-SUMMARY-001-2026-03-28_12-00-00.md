## Technical Implementation Details

### Architecture Decisions
1. **Shared detection logic via `run_detection()`:** Both API and web routes call the same function, preventing logic duplication and ensuring consistent behavior.
2. **Liveness vs readiness endpoints:** `/api/health/live` (always 200) for Docker healthcheck, `/api/health` (checks Presidio) for readiness probing.
3. **Volume-mounted source for development:** `docker-compose.yml` mounts `./src:/app/src` with `REDAKT_BASE_DIR` env var so templates/static resolve from the mount. Uvicorn `--reload` provides hot-reloading.
4. **Custom uvicorn log config:** Filters health check noise from access logs via `log_config.py`.

## Lessons Learned

### What Worked Well
1. Spec-driven implementation ensured complete coverage of requirements and edge cases
2. Critical review caught 2 HIGH bugs (allow_list integration, response_model union) before production
3. Shared `run_detection()` function eliminated logic duplication between API and web routes

### Challenges Overcome
1. **Presidio allow_list integration:** Initial implementation included bogus ad_hoc_recognizers. Verified against Presidio source code to fix.
2. **Docker volume mount vs installed package:** Templates loaded from site-packages instead of mounted source. Fixed with configurable `base_dir` setting.
3. **Uvicorn health check log noise:** Required custom log config dict (module-level filter insufficient with --reload).

### Recommendations for Future
- Keep spaCy config in the presidio fork (not mountable at build time due to model installation)
- When adding new features, extend `run_detection()` or create similar shared logic functions
- Integration tests should be added when CI/CD with Docker is configured
